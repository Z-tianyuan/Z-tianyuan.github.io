**问题 1：关于双重编码** 假设目标网站有一个极其严格的黑名单，拦截了 `admin` 这个词。为什么我们在 Burp Repeater 里手动修改包的时候，输入 `%61dmin` (单次编码) 会被拦截，而输入 `%2561dmin` (双重编码) 就能绕过？服务器内部究竟发生了什么？

**问题 2：关于开放重定向的本质** 在使用“开放重定向”绕过极其严格的本地路径限制时（比如限制必须以 `/product` 开头），为什么底层的 HTTP 请求库最终会去访问我们指定的内网 IP（`192.168.0.12`）？它为什么没有再次去问一遍安全过滤器“我能不能去”？

**问题 3：关于白名单的解析器差异** 在 `http://localhost%2523@stock.weliketoshop.net/` 这个 Payload 中，`@` 符号和 `%2523` (也就是 `#`) 分别承担了什么“欺骗”任务？它们分别骗过了谁？

期待你的硬核解答！答完之后，我们就可以看向截图里剩下那两个灰色的“带外检测的盲 SSRF (Blind SSRF)”了，那将是另一种完全不给你回显的暗杀技术。

我的理解：
问题1：这个是双重编码，第一次编码之后由于这个网站可以直接识别编码后的字符是什么样子的，所以一次编码之后这个服务器识别出来的就是原文，所以需要第二次的编码，


问题2：因为这个请求库中本身包含了一部分这个重定项，就是这个我找出的重定向本身就是这个本地服务器的白名单，所以它直接就放过去了，然后后面的那个指定的内网ip本身就是在内部的服务器里面去访问，所以额可以直接转到这个ip地址


问题3：这个@骗过的是那个看门的就是服务器对于url的限制，当这个服务器看到这个@后面的名字是本身就在这个白名单里面的，那么它就将这个请求给放过去了，然后进入内部服务器之后，这个#是用来对于那个检查名称的机制用来省略的，就是这个#后面的本省在内部是没有权限去干什么的，但是这个检查的机制有问题，它不看这个#号之后的东西，只看到这个前面的要求，然后就给你过去了


**✅ 针对问题 1（双重编码）：你抓住了“两次识别”的核心！**

- **你的理解：** 第一次编码被识别成原文，所以需要第二次编码。
    
- **专业术语升级：** 这叫 **WAF 与后端框架的解析差异**。
    
- **终极真相：** 门口的保安（WAF/安全过滤器）会自动进行**第一次 URL 解码**，看到 `%61dmin` 觉得没问题就放行了。然后，真正干活的业务部门（后端的 Spring/Node.js 等 Web 框架）在处理路由时，会自动进行**第二次 URL 解码**，把 `%61dmin` 还原成了致命的 `admin`。这就是经典的“套娃欺骗”。
    

**✅ 针对问题 2（开放重定向）：你抓住了“本地白名单+内部跳转”的核心！**

- **你的理解：** 重定向本身在白名单里，放过去之后，在内部服务器去访问指定的 IP。
    
- **专业术语升级：** 这叫 **HTTP 客户端的自动跟随跳转 (Auto-Redirect Follow)**。
    
- **终极真相：** 安全过滤器只检查了**你最初提交的那个 URL**（确实是合法的本地地址）。但是，当后端的 HTTP 库（比如 `curl` 或 `requests`）收到 302 重定向指令时，它有一个致命的默认习惯：**不向保安汇报，自己偷偷跟过去！** 过滤器根本不知道最终实际发起的网络连接去向了哪里。
    

**✅ 针对问题 3（白名单解析器差异）：完全命中！**

- **你的理解：** `@` 骗过了看门看名字的机制，`#` 让内部不去管后面的东西，只看前面的要求。
    
- **专业术语升级：** 这叫 **URL 解析器不一致性 (URL Parser Inconsistency) 导致的凭证注入 (Credential Injection) 与 锚点截断 (Fragment Truncation)**。
    
- **终极真相：** 你说得极其精准。保安（校验器）认为 `@` 后面是主机名；而网络司机（HTTP 发包器）认为 `#` 后面全是前端浏览器的注释废话，直接一刀切掉。这种由于底层代码库不同导致的认知撕裂，是白名单防御的绝对克星。
