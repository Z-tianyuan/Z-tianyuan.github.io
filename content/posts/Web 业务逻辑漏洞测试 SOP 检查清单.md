### Web 业务逻辑漏洞测试 SOP 检查清单

#### 步骤 1：业务流程拓扑与绘图 (Mapping & Recon)

逻辑漏洞不生在代码的语法里，而生在业务的流程里。

- **动作：** 正常走一遍核心业务流程（如：注册 -> 登录 -> 浏览商品 -> 加购物车 -> 结算 -> 支付 -> 退款）。

- **关键点：** 记录每一个步骤产生的 HTTP 请求，找出所有的关键参数（金额、数量、用户 ID、订单号、状态码）。

- **思考：** 开发者在这个环节，最怕发生什么事？（比如支付环节最怕少给钱，找回密码环节最怕发错人）。

#### 步骤 2：交易与支付风控绕过 (你已经掌握的部分 + 补充)

针对电商、充值等涉及钱的场景。

- **价格篡改：** 抓包拦截，将 `price` 或 `total_amount` 参数修改为 0.01 或极小值。

- **数量边界值溢出（你提到的）：** 将数量改为负数（如 `-1`）、小数（如 `0.5`），或者极大的数字（如 `9999999999`，看是否会导致后端整数溢出变成负数）。

- **⭐ 补充：步骤跳跃 (Skipping Steps)：** 获取正常支付成功后的页面 URL 或回调数据包。在下一次购物时，放弃支付，直接强行访问这个“成功页面”或向服务器发送“成功回调包”，看后端是否缺乏上下文校验，直接发货。

- **⭐ 补充：运费/优惠券逻辑：** 尝试叠加使用互斥的优惠券，或者把运费参数改成负数来抵扣商品价格。

#### 步骤 3：身份认证与凭证缺陷 (你已经掌握的部分 + 补充)

针对登录、注册、找回密码环节。

- **验证码暴力破解：** 在没有错误次数限制的情况下，使用 Burp Intruder 爆破 4 位或 6 位纯数字验证码。

- **凭证替换 / 狸猫换太子（你提到的）：** 用自己的手机/邮箱接收真实的验证码，在提交时抓包，将数据包中的 `username`、`phone` 或 `Session` 修改为目标受害者的，实现越权登录或密码重置。

- **⭐ 补充：验证码回显：** 抓取请求验证码的数据包，仔细观察服务器的响应包（Response）。有些粗心的开发者会直接把生成的验证码放在响应包里返回给前端。

- **⭐ 补充：空值绕过：** 在提交验证码时，尝试直接删除验证码参数（`code=`），或者将其改为 `null`、`false`，看后端是否默认放行。

#### 步骤 4：越权访问漏洞 (IDOR - 重点补充)

这是逻辑漏洞中最常见、也是危害最大的类型之一。

- **水平越权（看平级的数据）：** 你登录了账号 A，点开自己的订单，发现 URL 是 `?order_id=1001`。你把它改成 `1002`，如果能看到账号 B 的订单信息（甚至帮 B 取消订单），这就是水平越权。

- **垂直越权（普通用户变管理员）：** 抓取管理员在后台操作的数据包。用普通用户的 Session 去重放这个数据包。如果后端只在前端隐藏了管理菜单，但没有在接口层校验权限，普通用户就能执行管理员操作。

#### 步骤 5：并发与条件竞争 (Race Conditions - 重点补充)

利用服务器处理时间的微小差值。

- **场景：** 你只有 1 张无门槛 100 元优惠券。

- **动作：** 抓取使用优惠券的数据包，发送到 Burp Suite 的 Intruder，将 Payload 设置为 Null payloads（不改变数据），并将线程数拉满（比如 50 个并发请求），同时发送。

- **结果：** 正常的逻辑是使用 1 张后，数据库标记为已使用。但在极高并发下，数据库还没来得及更新状态，这 50 个请求可能都判断为“优惠券有效”，导致你用 1 张券抵扣了 50 次钱。

对于针对这种身份校验参数，我们会进行以下一系列尝试

1. **常规修改（篡改）：** 把最后一个字母 `t` 改成 `a`。
   
   - *预期：* 报错。说明系统确实在校验 token。

2. **留空测试（赋空值）：** 也就是你刚才做的，保留参数名，删掉值：`token=&username=carlos`。
   
   - *开发者逻辑漏洞：* 很多开发者写代码时会犯这种低级错误：
     
     Python
     
     ```
     if token: # 如果传了token，就去数据库校验
        verify(token)
     # 如果根本没传token的值（空值），代码直接跳过了校验块，往下执行了重置密码的逻辑！
     reset_password()
     ```

3. **彻底删除（去参数）：** 把 `temp-forgot-password-token=` 这一整串从请求包里彻底删掉，只发 `username=carlos...`。
   
   - *效果：* 如果开发者的代码没做空指针异常处理，有时候这会引发系统报错，甚至直接绕过验证。

4. **类型转换（数组绕过）：** 把参数名改成 `token[]=xxx`。这在 PHP 写的网站里特别管用，有时能引发后端的致命错误或直接绕过。

5. **参数污染（双重参数）：** 发送 `token=你的真实token&token=随便乱写的token`。看看服务器是读取第一个，还是读取第二个。

**黑客的直觉是怎么来的？** 就是靠这一个个失败的测试积累出来的。你尝试了 1、3、4、5 都失败了（返回 200 OK 并报错），但当你尝试了第 2 步（留空），突然收到了一个美妙的 `302 Found`。那一瞬间，你就知道：“哈！写这行代码的程序员偷懒了！”
