这是一个极其经典的 **“开发语言与操作系统的解析差异”**！也是黑客最喜欢利用的盲区。

1. **保安（黑名单过滤器）的死板逻辑：** 程序员写了一段简单的代码来防御 SSRF：“如果用户输入的 URL 里包含了 `127.0.0.1` 或者 `localhost` 这几个具体的**字符串**，我就拦截（400 报错）。”
    
2. **快递员（底层操作系统）的聪明补全：** 当你输入 `http://127.1/` 时，保安看了一眼，心里想：“嗯，没有 `127.0.0.1` 这个字符串，放行！” 但是，当这个请求交到底层操作系统（比如 Linux 的网络组件）去实际发送网络请求时，**操作系统的网络协议栈极其聪明（或者说极其宽容）**。它规定：如果一个 IP 地址缺少了中间的段，它会**自动用 0 帮你补齐**！
    
    - 你输入 `127.1`，系统自动补全为 `127.0.0.1`
        
    - 你输入 `10.1`，系统自动补全为 `10.0.0.1`
        

**结论：** 你利用了字符串匹配的死板，骗过了第一层保安，然后利用了底层的自动补全，成功让服务器访问了自己！




我们来看看你的终极 Payload 在服务器内部引发了怎样的“认知分裂”： 
**Payload:** `http://localhost:80%2523@stock.weliketoshop.net`

_(注：`%2523` 就是 `#` 的双重 URL 编码，这和你上一个实验写 Python 脚本对 `a` 进行双重编码的原理一模一样。系统解码一次后，它变成了 `%23`)_

**保安的视角（校验白名单）：** 保安拿着 URL：`http://localhost:80%23@stock.weliketoshop.net` 保安的逻辑比较死板，他想：“我知道 `@` 符号前面是账号密码，后面才是真正的主机名。所以我直接看 `@` 后面的内容就行了。” 保安一看 `@` 后面是 `stock.weliketoshop.net`，完美符合白名单！ **保安判定：安全放行！**

**司机的视角（发起真正请求）：** 司机接过了保安放行的 URL。但是司机的底层库更聪明（或者说更复杂）。 司机看到 URL 里有一个 `%23` (或者被进一步解码成了 `#`)。 司机心想：“哎呀，这有个 `#` 号！网络协议规定 `#` 后面的东西全是没用的注释（Fragment），我统统不要了！” 于是，司机直接把 `#` 以及它后面所有的东西（包括那个 `@stock.weliketoshop.net`）全部**砍掉**了。 司机真正发起的请求地址变成了：`http://localhost:80` **司机判定：全速开往本地管理员控制台！**

**结论：你用 `@` 骗过了保安，用 `#` 砍断了司机的方向盘！**
