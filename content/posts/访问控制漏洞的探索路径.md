##### 🗺️ 访问控制/越权狩猎标准路径 (SOP)

#### 📍 第一步：建立“多重身份”观测站 (The Setup)

在挖掘越权之前，你必须拥有对比参照物。不要只拿一个账号瞎点。

1. **注册账号 A（攻击者）：** 比如 `attacker:123`。

2. **注册账号 B（受害者）：** 比如 `victim:123`。

3. **寻找高权限入口：** 如果网页上有公开的管理员账号密码（如 `administrator:admin`），也登录它备用。

4. *操作：打开三个不同的浏览器（或使用隐私模式/多账号插件），分别登录这三个账号。*

#### 📍 第二步：雷达扫描，锁定“敏感标识符” (Mapping & Recon)

操作账号 A，在网站里像正常用户一样闲逛：修改资料、下订单、看账单、发私信。
此时，眼睛死死盯着 Burp Suite 的 **HTTP History**，寻找以下特征的参数：

- **数字 ID：** `?user_id=1234` 或 `{"order_id": 5678}` (最经典的 IDOR 目标)。

- **用户名：** `/profile/carlos` 或 `?account=wiener`。

- **文件路径：** `?file=receipt_101.pdf`。

- **角色标识：** `{"role": "user"}` 或 `{"isAdmin": false}` (纵向越权的高价值目标)。

#### 📍 第三步：发动“横向降维打击” (Horizontal IDOR Test)

一旦你在账号 A 的请求包里看到了代表身份的 ID（比如 `id=101`），立刻送到 **Repeater**。

1. **直接替换：** 把 `id=101` 改成账号 B 的 `id=102`。

2. **发包观察：**
   
   - 如果返回 `200 OK` 并且你看到了账号 B 的敏感信息（比如收货地址、私密账单） -> **横向越权读取，成功！**
   
   - 如果你修改的是一个 `POST /delete_message` 请求，把消息 ID 改成 B 的，发送后返回 `200` 且消息真被删了 -> **横向越权操作，高危！**
   
   - 如果返回 `403 Forbidden` 或 `401 Unauthorized` -> *防御有效，换下一个接口。*

#### 📍 第四步：发动“纵向权力篡夺” (Vertical Escalation Test)

这是挑战系统底层架构的时候。

1. **强行盲猜 (Forced Browsing)：** 账号 A 登录状态下，直接在 URL 栏输入后台地址，如 `/admin`, `/administrator`, `/admin_panel`, `/admin/deleteUser?username=carlos`。看看系统是直接让你进，还是把你踢出来。

2. **偷天换日 (Parameter Tampering)：** 在账号 A 的资料更新请求、或者注册请求中，强行塞入一些你猜测的特权参数。比如在 JSON 里加上 `"roleid": 1` 或 `"admin": true`，看看后端是否会照单全收。

3. **借尸还魂 (Cookie 欺骗)：** 如果你有一个普通 Session 和一个 Admin Session。尝试把 Admin 专属请求（比如删用户）抓下来，然后把里面的 Session 换成普通用户的，发过去看看系统认不认。

### 💡 终极防卡关提示（The Hacker's Intuition）

如果你试了上面的方法，怎么都改不成功（比如 ID 是像 `f8a9e...` 这样猜不出的 UUID 乱码，或者一直报 403），**不要马上看教程，试着思考以下两个方向：**

- **信息泄露在哪？** 既然 UUID 猜不到，那系统一定在某个不起眼的地方（比如发帖的作者信息里、或者前端的 JavaScript 变量里）把受害者的 UUID 泄露给你了。去找！

- **是不是前端在骗我？** 很多时候，纵向越权返回了 403，但其实后端已经把操作执行了，只是不让你看结果；或者，你直接在响应包里把 `HTTP/1.1 403 Forbidden` 改成 `HTTP/1.1 200 OK`，就能看到被隐藏的前端管理员面板！
