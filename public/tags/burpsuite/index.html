    Web 业务逻辑漏洞测试 SOP 检查清单#

    步骤 1：业务流程拓扑与绘图 (Mapping & Recon)#

    逻辑漏洞不生在代码的语法里，而生在业务的流程里。

    • 动作： 正常走一遍核心业务流程（如：注册 -> 登录 -> 浏览商品 -> 加购物车 -> 结算 -> 支付 -> 退款）。

    • 关键点： 记录每一个步骤产生的 HTTP 请求，找出所有的关键参数（金额、数量、用户 ID、订单号、状态码）。

    • 思考： 开发者在这个环节，最怕发生什么事？（比如支付环节最怕少给钱，找回密码环节最怕发错人）。

    步骤 2：交易与支付风控绕过 (你已经掌握的部分 + 补充)#

    针对电商、充值等涉及钱的场景。

    • 价格篡改： 抓包拦截，将 price 或 total_amount 参数修改为 0.01 或极小值。

    • 数量边界值溢出（你提到的）： 将数量改为负数（如 -1）、小数（如 0.5），或者极大的数字（如 9999999999，看是否会导致后端整数溢出变成负数）。

    • ⭐ 补充：步骤跳跃 (Skipping Steps)： 获取正常支付成功后的页面 URL 或回调数据包。在下一次购物时，放弃支付，直接强行访问这个“成功页面”或向服务器发送“成功回调包”，看后端是否缺乏上下文校验，直接发货。

    • ⭐ 补充：运费/优惠券逻辑： 尝试叠加使用互斥的优惠券，或者把运费参数改成负数来抵扣商品价格。

    步骤 3：身份认证与凭证缺陷 (你已经掌握的部分 + 补充)#

    针对登录、注册、找回密码环节。

    • 验证码暴力破解： 在没有错误次数限制的情况下，使用 Burp Intruder 爆破 4 位或 6 位纯数字验证码。

    如何训练自己打破“固定思维”

    March 29, 2026
    漏洞挖掘
    SQL注入, 靶场实战, BurpSuite

    🧠 如何训练自己打破“固定思维”？#

    要改掉做数学题的习惯，你需要刻意练习**“状态机模糊测试（State Machine Fuzzing）”**的思维。以后遇到任何复杂的业务逻辑，严格按照这三步走：

    第一步：全流程摸底（先做个好人）#

    绝对不要一上来就抓包改数据！

    1. 注册一个正常账号。

    2. 买一件最便宜的东西，老老实实付钱。

    3. 记录下这个过程中产生的每一个核心 URL 和参数。（比如：/cart 👉 /checkout 👉 /order-confirmation）。这就相当于你拿到了迷宫的完整地图。

    第二步：寻找“状态依赖”（灵魂拷问）#

    看着你记录下来的流程地图，问自己一个问题：“步骤 3 怎么知道我已经完成了步骤 2？”

    • 安全的系统： 步骤 3 的代码会去查数据库：“这个订单的状态是 PAID 吗？如果是，才展示成功页面。”

    • 脆弱的系统（比如这个实验室）： 程序员偷懒了。他认为“只有付完款的网页，才会跳转到步骤 3 的 URL”。所以步骤 3 的代码什么都不查，只要有人访问这个 URL，它就默认这人付过钱了！

    第三步：打乱顺序（实施降维打击）#

    你已经知道终点在哪里了。现在，回到起点，把那件天价皮夹克放进购物车（步骤 1）。 然后，不理会网页上的“结账”按钮，直接在 Burp Suite 里，强制向服务器发送步骤 3 的 URL。

    你问出了一个渗透测试工程师走向成熟的终极问题。

    照着教程把漏洞复现出来，那叫“漏洞验证”；但在一个完全未知的、没有任何提示的黑色盒子里把漏洞揪出来，那才叫真正的**“漏洞挖掘 (Bug Hunting)”**。

    当你以后在真实的漏洞赏金平台（比如 HackerOne 上的企业级目标）进行实战挖掘时，你是绝对碰不到这种给你写好前提条件和步骤的靶场的。面对一个复杂的企业级系统，黑客寻找这类**“状态机逻辑缺陷 (Flawed State Machine)”**漏洞，有一套极其严密的标准流程（SOP）。

    这是一套被称为**“业务流逆向降维分析”**的实战 WBS。我为你总结了四个阶段：

    🕵️‍♂️ 状态机漏洞发现 SOP (标准作业程序)#

    第一阶段：全量状态测绘 (Mapping the Happy Path)#

    不要一上来就想着搞破坏！黑客的第一步是把自己伪装成一个极其听话的“好用户”。

    1. 清空历史： 在 Burp 的 HTTP History 中打一个标记（或者清空之前的记录），确保接下来的流量是纯净的。

    2. 顺流而下： 关闭拦截 (Intercept is off)。在浏览器中，极其缓慢且完整地走完整个业务流程（比如：输入账号密码 -> 点击登录 -> 页面跳转 -> 选择角色 -> 进入主页）。

    3. 绘制拓扑图： 回到 Burp，把刚才这套流程产生的所有核心 URL 提取出来，在脑子里（或纸上）画一条线： A (/login) ➡️ B (/role-selector) ➡️ C (/home)

    第二阶段：锚定“权力交接点” (Identify State Transitions)#

    这是最考验深度思考的一步。你需要看着 HTTP History 里的数据包，找出系统是在哪一刻决定你的权限的。

    1. 追踪 Cookie/Token 的诞生： 仔细看 A (/login) 的响应包。它是不是在这里通过 Set-Cookie 给你发了身份证？

    🧠 第一部分：漏洞的核心原理#

    这个漏洞之所以能成立，是因为目标网站的后端系统存在**“精神分裂”**——处理注册流程的两个核心组件，对数据的校验标准不一致。

    1. 宽容的“发件员”（邮件处理服务）#

    当你在网页上提交注册邮箱时，负责发送验证邮件的代码逻辑非常简单粗暴。它只认死理：“只要这是一个格式合法的邮箱，我就往它 @ 后面的那个顶级域名发邮件。” 它没有限制长度。所以当你提交 238个a@dontwannacry.com.你的实验域名.net 时，它一看后缀是你的实验域名，就乖乖把邮件发到了你的客户端里。

    2. 严格且粗暴的“档案管理员”（数据库）#

    发件员发完邮件后，要把你的信息存入后台数据库（比如 MySQL 的 users 表）。 在设计数据库时，程序员给 email 这个字段设定了最大长度，通常是 VARCHAR(255)。

    当超过 255 个字符的数据涌进来时，老版本的数据库（或者未开启严格模式的数据库）不会报错拦截，而是会采取一种极其粗暴的默认行为：“既然装不下，那我就把第 255 个字符后面的所有东西，全部拿剪刀咔嚓剪掉！”

    3. 致命的“身份核验”（权限逻辑）#

    网站的权限判断逻辑是：用户登录后，去数据库里查他的 email 字段，如果以 @dontwannacry.com 结尾，就给他 Admin 权限。

    💥 漏洞闭环： 你用超长邮箱骗过了“发件员”拿到了激活链接；随后“档案管理员”挥舞剪刀，把你邮箱后缀的实验域名剪掉了，只把前 255 个字符（刚好以 @dontwannacry.com 结尾）存进了数据库。 当你登录时，权限系统一看数据库里的记录，当场肃然起敬：“欢迎您，内部管理员！”

    🕵️‍♂️ 第二部分：如何挖掘出这个漏洞？（黑客视角的详细过程）#

    你可能会问：“原版教程是直接把答案喂给我了，但如果这是一个完全陌生的网站，我怎么知道要往框里填 255 个字符？”

    真正的挖掘过程，是一个严密的**“假设 -> 测试 -> 观察 -> 利用”**的循环。以下是标准的渗透测试思路：

    Step 1: 确立目标与信息收集 (侦察阶段)#

    • 观察现象： 你在网站上随便乱点，可能在某处（比如 JS 文件、或者是之前的目录扫描）发现了一个 /admin 路径。

    • 触碰边界： 你尝试访问 /admin，网页弹出一个无情的错误提示：Admin interface only available if logged in as a @dontwannacry.com user。

    进阶版的 SQL 注入标准化操作路径 (SOP)#

    为了让你在脱离教程后不再迷茫，你需要把上面的思路重组成一个无死角的测试漏斗。遇到任何网站，按照这5步走：

    步骤 1：全方位寻找注入点 (信息收集)#

    不要只盯着网页上的输入框或URL。在Burp Suite里，任何由客户端传给服务器的数据都可能是注入点。

    • 显性输入： URL参数 (GET)、表单提交 (POST)。

    • 隐性输入： HTTP请求头（如 User-Agent、Referer）、Cookie 的值。

    步骤 2：测试闭合与判断数据类型 (Fuzzing)#

    找到参数后，先不要急着写恶意代码，先破坏它原有的SQL语句结构，看看网站会不会“报错”或“异常”。

    • 数字型还是字符型？

      • 输入 id=1' 或 id=1" 或 id=1\: 观察页面是否报错或内容消失。这就叫“寻找闭合符”。

      • 输入 id=1 and 1=1 (正常) 和 id=1 and 1=2 (异常): 如果表现不同，说明存在数字型注入。

    步骤 3：确定数据库类型与注入策略 (定级)#

    根据网站在第二步的反馈，选择最有效的注入手法，优先级从高到低：

    1. 联合查询注入 (Union Based)： 页面能直接显示数据库里的数据内容（最简单直接）。

    2. 报错注入 (Error Based)： 页面虽然不显示正常数据，但会把 SQL 语法报错的详细信息（甚至带出数据）吐在页面上。

    3. 布尔盲注 (Boolean Blind)： 页面不报错，也不显示数据，但“输入对的逻辑”和“输入错的逻辑”，页面的长相或状态码不一样。

    4. 时间盲注 (Time-based Blind)： 无论输入对错，页面长得一模一样。只能通过加入 sleep(5) 这样的指令，看服务器是不是真的卡了5秒才响应来判断注入是否存在（最耗时，难度最高）。
