    进阶版的 SQL 注入标准化操作路径 (SOP)#

    为了让你在脱离教程后不再迷茫，你需要把上面的思路重组成一个无死角的测试漏斗。遇到任何网站，按照这5步走：

    步骤 1：全方位寻找注入点 (信息收集)#

    不要只盯着网页上的输入框或URL。在Burp Suite里，任何由客户端传给服务器的数据都可能是注入点。

    • 显性输入： URL参数 (GET)、表单提交 (POST)。

    • 隐性输入： HTTP请求头（如 User-Agent、Referer）、Cookie 的值。

    步骤 2：测试闭合与判断数据类型 (Fuzzing)#

    找到参数后，先不要急着写恶意代码，先破坏它原有的SQL语句结构，看看网站会不会“报错”或“异常”。

    • 数字型还是字符型？

      • 输入 id=1' 或 id=1" 或 id=1\: 观察页面是否报错或内容消失。这就叫“寻找闭合符”。

      • 输入 id=1 and 1=1 (正常) 和 id=1 and 1=2 (异常): 如果表现不同，说明存在数字型注入。

    步骤 3：确定数据库类型与注入策略 (定级)#

    根据网站在第二步的反馈，选择最有效的注入手法，优先级从高到低：

    1. 联合查询注入 (Union Based)： 页面能直接显示数据库里的数据内容（最简单直接）。

    2. 报错注入 (Error Based)： 页面虽然不显示正常数据，但会把 SQL 语法报错的详细信息（甚至带出数据）吐在页面上。

    3. 布尔盲注 (Boolean Blind)： 页面不报错，也不显示数据，但“输入对的逻辑”和“输入错的逻辑”，页面的长相或状态码不一样。

    4. 时间盲注 (Time-based Blind)： 无论输入对错，页面长得一模一样。只能通过加入 sleep(5) 这样的指令，看服务器是不是真的卡了5秒才响应来判断注入是否存在（最耗时，难度最高）。

    步骤 4：层层剥茧提取数据 (利用)#

    确定了策略后，严格按照这个顺序拼接Payload获取信息：

    1. 查当前数据库名 database() 和用户 user()。

    2. 查该数据库里所有的表名 (information_schema.tables)。

    3. 查目标表里的所有列名/字段名 (information_schema.columns)。

    4. 最后才是提取具体字段的数据（如 username, password）。

    步骤 5：遇到拦截时的绕过 (WAF Bypass)#

    如果发现提交某个Payload被拦截（如弹出“非法输入”或 403 页面），才进入你提到的绕过环节。替换空格为 /**/、对关键字进行双重URL编码、使用等价函数代替被封杀的函数等。
